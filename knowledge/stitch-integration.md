# Google Stitch Integration

Canonical reference for Google Stitch AI UI generation within Integra Developer Studio. All agents and skills should read this file when working with Stitch.

---

## 1. Overview

Google Stitch is an AI UI design tool that generates multi-screen web UIs from natural language descriptions. It was launched at Google I/O 2025 (from the Galileo AI acquisition) and is available free with a Google account.

Within Integra Developer Studio, Stitch is used as a **visual specification tool** -- it outputs static HTML, NOT production React components. The generated HTML serves as a design reference for layout, spacing, color usage, and visual hierarchy. The executor agent rebuilds the UI in Next.js/React using shadcn/ui + Tailwind.

Stitch is integrated via:
- **MCP server:** `@_davideast/stitch-mcp` (provides 3 tools to Claude Code agents)
- **SDK package:** `@google/stitch-sdk` v0.0.3 (for programmatic access when needed, e.g., `createDesignSystem()`)

Stitch is **purely additive** -- the entire build pipeline works without it. If unavailable, agents fall back to manual UI design workflows.

## 2. MCP Server Setup

Add the Stitch MCP server to the project root `.mcp.json`:

```json
{
  "mcpServers": {
    "stitch": {
      "command": "npx",
      "args": ["@_davideast/stitch-mcp", "proxy"],
      "env": {
        "STITCH_API_KEY": "${STITCH_API_KEY}"
      }
    }
  }
}
```

**Requirements:**
- The `STITCH_API_KEY` environment variable must be set before the MCP server starts
- Get the key from https://stitch.withgoogle.com/ (Google account required)
- The MCP server must be configured in the **project root** `.mcp.json`, not in a subdirectory
- The `@_davideast/stitch-mcp` package is fetched via `npx` on first run -- no pre-install needed

## 3. MCP Tools Reference

The Stitch MCP server exposes exactly 3 tools:

| Tool | Purpose | Key Parameters | Returns |
|------|---------|---------------|---------|
| `build_site` | Builds a site from a Stitch project, mapping screens to routes | Project description (natural language), screen-to-route mapping (e.g., `{ "/": "Dashboard", "/trade": "Trade" }`) | Site structure with screen IDs for each generated screen |
| `get_screen_code` | Retrieves the HTML/CSS for a single generated screen | Screen ID (from `build_site` response) | Raw HTML string containing the full markup and inline/embedded CSS |
| `get_screen_image` | Retrieves a screenshot of a generated screen as a base64-encoded image | Screen ID (from `build_site` response) | Base64-encoded PNG image suitable for visual review |

**Usage pattern:**
1. Call `build_site` first to generate the site and receive screen IDs
2. Call `get_screen_image` to visually review screens before exporting
3. Call `get_screen_code` to export the HTML for selected screens

All three tools are available to the `frontend-designer` agent as `mcp__stitch__build_site`, `mcp__stitch__get_screen_code`, and `mcp__stitch__get_screen_image`.

## 4. Design System Creation

Use the `createDesignSystem()` SDK pattern to maintain visual consistency across Stitch generations.

### When user selected "Integra official branding"

Create a Stitch design system from the official Integra brand tokens (see `knowledge/design-systems/integra-brand.md`):

- **Primary color:** Coral `#FF6D49`
- **Primary dark:** `#FC4E23` (hover/active)
- **Primary light:** `#FF8A65` (subtle accents)
- **Secondary accent:** Pink `#F34499`
- **Font:** Euclid Circular B (400/500/600/700)
- **Code font:** Geist Mono
- **Background:** `#0A0A0F` (dark theme)
- **Card surface:** `rgba(23, 23, 28, 0.7)`
- **Text:** `#E6E6E6` (primary), `#A3A3A3` (muted), `#6C757D` (subtle)
- **Gradients:** Brand gradient `linear-gradient(135deg, #FFAFD6, #F34499, #FC4E23, #F71227)`
- **Corners:** 12px cards, 6px buttons
- **Glassmorphism:** `backdrop-filter: blur(12px)` + `rgba(255,255,255,0.05)` background

### When user selected "custom branding"

Create from the tokens in `knowledge/design-systems/custom-brand.md`, which is generated by the `ui-ux-pro-max` skill during the UI pipeline. The custom brand file follows the same structure but with user-specific palette, typography, and spacing.

### How design systems are applied

The design system description is passed as **context to every `build_site` call** to maintain visual consistency. Example prompt structure:

```
Build a [dApp concept] dashboard with the following design system:
- Primary: #FF6D49 (Coral), Dark surfaces (#0A0A0F bg), Euclid Circular B font
- Cards: semi-transparent dark surfaces with 12px rounded corners
- Style: glassmorphism with subtle blur effects
```

**LIMITATION:** Stitch has no persistent design system enforcement across generations. Each generation can drift from the specified design system. **Mitigation:** Pass the full design system description in every prompt. Do not rely on Stitch "remembering" previous generations' styles.

## 5. Variant Generation Workflow

Generate multiple visual variants per screen so the user can choose their preferred style.

### Steps

1. **Generate base site:** The frontend-designer agent calls `build_site` with the dApp concept + design system description, mapping screens to routes
2. **Generate 1-3 variants:** Call `build_site` again with different style prompts for the same screen layout:
   - Variant A: "minimal, clean, lots of whitespace"
   - Variant B: "bold, high contrast, prominent CTAs"
   - Variant C: "data-dense, compact tables, information-rich"
3. **Capture screenshots:** Use `get_screen_image` to capture a base64 PNG screenshot of each variant
4. **Present to user:** The build skill presents the screenshots to the user via `AskUserQuestion` for selection (e.g., "Which style do you prefer? 1. Minimal 2. Bold 3. Data-dense")
5. **Export selected:** Use `get_screen_code` to export the HTML of the user's chosen variant

### Rules

- Always generate at least 1 variant; up to 3 for key screens (dashboard, main action page)
- Secondary screens (settings, about) can use a single generation matching the selected style
- Each variant call counts against the monthly generation limit (see Section 8)
- The user selection happens in the **build skill** (not in a subagent), since `AskUserQuestion` only works in the top-level skill context

## 6. HTML Export Pipeline

Stitch HTML becomes Next.js components through a structured conversion process. This is the most critical section -- it defines the boundary between Stitch output and production code.

### Pipeline

1. **Stitch exports raw HTML/CSS** -- static markup with no interactivity, no React, no state management. Forms do not submit. Buttons do not navigate. It is a visual mockup.
2. **Frontend-designer agent includes the HTML** in its Frontend Design Document under a "Stitch Reference HTML" section for each screen
3. **Executor agent uses the HTML as a VISUAL SPECIFICATION** -- extracting layout structure, spacing relationships, color usage, and component hierarchy
4. **Executor REBUILDS components** in Next.js/React using:
   - shadcn/ui components as the base
   - Tailwind CSS for styling (using Integra design system tokens)
   - Proper React state management, event handlers, and routing
   - wagmi v2 + viem for blockchain interactions

### Rules (CRITICAL)

- **NEVER** use `dangerouslySetInnerHTML` with Stitch output
- **NEVER** copy-paste Stitch HTML directly into JSX files
- **NEVER** ship Stitch HTML as production code
- The HTML is a **design reference**, not code to deploy
- Stitch CSS classes should be translated to Tailwind equivalents
- Stitch layout structure should inform component decomposition
- Stitch color values should be replaced with Integra design system CSS variables

### What the executor extracts from Stitch HTML

- Page layout grid (sidebar width, content area, header height)
- Card/section arrangement and visual hierarchy
- Spacing patterns (padding, margins, gaps between elements)
- Typography scale in use (heading sizes, body text, captions)
- Color application patterns (which surfaces use which shades)
- Component placement (where buttons, inputs, tables appear)

## 7. Graceful Fallback

Stitch is purely additive. The entire Integra Developer Studio pipeline works without it.

### When `STITCH_API_KEY` is not set or MCP server is not configured

- The frontend-designer agent **skips all Stitch MCP calls**
- Falls back to its standard workflow: component tree + wireframe descriptions
- The build skill logs: `"Stitch unavailable -- using manual UI pipeline"`
- **No error is thrown** -- the pipeline continues normally
- The user still gets a complete Frontend Design Document, just without Stitch reference HTML

### When Stitch API returns an error mid-generation

- Log the error with the specific screen that failed
- Skip that screen's Stitch output
- Continue generating remaining screens
- Fallback screens get manual component tree descriptions from the frontend-designer agent
- The build skill logs which screens have Stitch HTML and which use manual descriptions

### When the wizard detects Stitch is unavailable

- If `STITCH_API_KEY` is not in the environment, the wizard should not offer "Generate UI with Stitch AI" as an option
- If Stitch becomes unavailable after the wizard (e.g., API key expires), the build skill falls back gracefully mid-pipeline

## 8. Generation Limits

Google Stitch enforces monthly generation limits on the free tier:

| Tier | Limit | Notes |
|------|-------|-------|
| Standard generations | 350 per month | Regular `build_site` calls |
| Experimental generations | 200 per month | Advanced features, newer models |

- There is **no paid tier dependency** -- Integra Studio uses free tier only
- Each `build_site` call (including variant generations) counts as one generation
- `get_screen_code` and `get_screen_image` do NOT count against limits (they retrieve existing output)
- If limits are hit mid-project, the build skill falls back to the manual UI pipeline for remaining screens
- The fallback is transparent to the user -- they still get a complete design, just without Stitch visuals for some screens

### Budget planning

For a typical 5-screen dApp with 3 variants per key screen:
- 3 key screens x 3 variants = 9 generations
- 2 secondary screens x 1 generation = 2 generations
- **Total: ~11 generations per project** (well within monthly limits)

## 9. Environment Variables

| Variable | Required | Description |
|----------|----------|-------------|
| `STITCH_API_KEY` | For Stitch features | API key for Google Stitch. Get from https://stitch.withgoogle.com/ |

### Setup

1. Visit https://stitch.withgoogle.com/ and sign in with a Google account
2. Navigate to API settings and generate an API key
3. Add to your `.env` file: `STITCH_API_KEY=your_key_here`
4. **Never commit `.env`** -- it must be in `.gitignore`
5. Add to `.env.example` with a placeholder:

```bash
# Google Stitch AI UI Generation (optional -- UI works without it)
# Get your key: https://stitch.withgoogle.com/
STITCH_API_KEY=
```

## 10. Known Limitations

| Limitation | Impact | Mitigation |
|------------|--------|------------|
| **Static HTML only** | No React components, no interactivity, no state management | Treat as design reference; executor rebuilds in Next.js |
| **No persistent design system** | Each generation can drift from the specified design tokens | Pass full design system description in every `build_site` prompt |
| **Google Labs product** | May be deprecated or changed without notice | Design the pipeline to work without Stitch (graceful fallback) |
| **Generation limits** | 350 standard / 200 experimental per month caps heavy iteration | Budget generations per project (~11 per 5-screen dApp); fallback to manual for overflow |
| **No React export** | Stitch does not export React/JSX (React export rumored for future) | Manual conversion via executor agent using Stitch HTML as visual spec |
| **Font limitations** | Stitch may not have access to Euclid Circular B (commercial font) | Accept font substitution in Stitch output; executor applies correct font in Next.js build |
