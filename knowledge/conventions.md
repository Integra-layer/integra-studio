Standard coding conventions for building dApps on the Integra blockchain. Follow these for consistency across the ecosystem.

## Solidity Conventions

### File Naming

- Contract files: `PascalCase.sol` (e.g., `AssetManager.sol`, `RewardDistributor.sol`)
- Interface files: `IPascalCase.sol` in `contracts/interfaces/` (e.g., `IAssetManager.sol`)
- Library files: `PascalCase.sol` in `contracts/libraries/` (e.g., `MathUtils.sol`)
- Test files: `ContractName.test.ts` in `test/` (e.g., `AssetManager.test.ts`)

### Naming Conventions

```solidity
// Contracts: PascalCase
contract AssetManager { ... }

// Interfaces: I-prefix + PascalCase
interface IAssetManager { ... }

// Libraries: PascalCase
library MathUtils { ... }

// Functions: camelCase
function mintPassport(...) external { ... }

// Public/external state variables: camelCase
uint256 public totalSupply;

// Private/internal state variables: _underscoreCamelCase
uint256 private _counter;
mapping(address => uint256) internal _balances;

// Constants: UPPER_SNAKE_CASE
uint256 public constant MAX_SUPPLY = 10_000;
uint256 private constant _FEE_BASIS_POINTS = 250;

// Events: PascalCase (past tense where applicable)
event PassportMinted(uint256 indexed tokenId, address indexed to);

// Errors: PascalCase with contract prefix
error AssetManager__NotOwner();
error AssetManager__InvalidAmount(uint256 amount);

// Modifiers: camelCase
modifier onlyIssuer(uint256 tokenId) { ... }

// Enums: PascalCase
enum OrderStatus { Open, Filled, Cancelled }

// Structs: PascalCase
struct PassportData { ... }
```

### Pragma and Imports

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20; // Lock to 0.8.x, prefer latest stable

// Imports: named imports, alphabetical order
import {ERC721} from "@openzeppelin/contracts/token/ERC721/ERC721.sol";
import {Ownable} from "@openzeppelin/contracts/access/Ownable.sol";
import {ReentrancyGuard} from "@openzeppelin/contracts/utils/ReentrancyGuard.sol";
import {IAssetManager} from "./interfaces/IAssetManager.sol";
```

### NatSpec Documentation

Every public and external function MUST have NatSpec:

```solidity
/// @notice Mint a new asset passport for the given recipient
/// @dev Increments the token counter and stores passport metadata on-chain
/// @param to The address that will own the passport
/// @param name Human-readable name for the asset
/// @param metadataURI IPFS or HTTP URI pointing to full metadata JSON
/// @return tokenId The ID of the newly minted passport
function mintPassport(
    address to,
    string calldata name,
    string calldata metadataURI
) external returns (uint256 tokenId) {
    // ...
}
```

### Function Ordering

Within a contract, order functions as follows:
1. Constructor
2. Receive/fallback (if any)
3. External functions
4. Public functions
5. Internal functions
6. Private functions
7. View/pure functions (within each visibility group, put them last)

### Security Patterns

```solidity
// ALWAYS: Checks-Effects-Interactions pattern
function withdraw(uint256 amount) external nonReentrant {
    // 1. Checks
    if (amount == 0) revert InvalidAmount(amount);
    if (_balances[msg.sender] < amount) revert InsufficientBalance();

    // 2. Effects (state changes BEFORE external calls)
    _balances[msg.sender] -= amount;

    // 3. Interactions (external calls LAST)
    (bool success, ) = msg.sender.call{value: amount}("");
    if (!success) revert TransferFailed();
}

// ALWAYS: Use custom errors instead of require strings
error InvalidAmount(uint256 amount);
error InsufficientBalance();

// ALWAYS: Use ReentrancyGuard for functions with external calls
// ALWAYS: Validate inputs (zero address, zero amount, bounds)
// ALWAYS: Emit events for all state changes
// NEVER: Use tx.origin for authentication
// NEVER: Use selfdestruct
// NEVER: Leave functions without access control if they modify state
```

### Testing Patterns

```typescript
// test/AssetManager.test.ts
import { expect } from "chai";
import { ethers } from "hardhat";
import { loadFixture } from "@nomicfoundation/hardhat-toolbox/network-helpers";

describe("AssetManager", function () {
  // Shared fixture for all tests
  async function deployFixture() {
    const [owner, user1, user2] = await ethers.getSigners();
    const AssetManager = await ethers.getContractFactory("AssetManager");
    const contract = await AssetManager.deploy();
    return { contract, owner, user1, user2 };
  }

  describe("mintPassport", function () {
    it("should mint a passport to the recipient", async function () {
      const { contract, user1 } = await loadFixture(deployFixture);
      await expect(contract.mintPassport(user1.address, "Test", "uri://test"))
        .to.emit(contract, "PassportMinted")
        .withArgs(1, user1.address);
    });

    it("should revert when recipient is zero address", async function () {
      const { contract } = await loadFixture(deployFixture);
      await expect(contract.mintPassport(ethers.ZeroAddress, "Test", "uri://test"))
        .to.be.revertedWithCustomError(contract, "InvalidAddress");
    });

    it("should increment token IDs sequentially", async function () {
      const { contract, user1 } = await loadFixture(deployFixture);
      await contract.mintPassport(user1.address, "A", "uri://a");
      await contract.mintPassport(user1.address, "B", "uri://b");
      expect(await contract.ownerOf(2)).to.equal(user1.address);
    });
  });
});
```

Test every:
- Happy path (function works as expected)
- Revert cases (invalid inputs, unauthorized callers)
- Edge cases (zero values, max values, boundary conditions)
- Event emissions
- State changes

## TypeScript / React Conventions

### File Naming

- Components: `kebab-case.tsx` (e.g., `asset-card.tsx`, `xp-badge.tsx`)
- Hooks: `camelCase.ts` with `use` prefix (e.g., `useAssetPassport.ts`)
- Utilities: `kebab-case.ts` (e.g., `format-address.ts`)
- Types: `kebab-case.ts` or `index.ts` in types/ directory
- Pages: `page.tsx` inside app router directories (Next.js convention)
- Layouts: `layout.tsx` inside app router directories

### Naming Conventions

```typescript
// Components: PascalCase
export function AssetCard({ asset }: AssetCardProps) { ... }

// Hooks: camelCase with use prefix
export function useAssetPassport(tokenId: bigint) { ... }

// Types/Interfaces: PascalCase
interface AssetCardProps {
  asset: PassportData;
  onSelect?: (tokenId: bigint) => void;
}

type OrderStatus = "open" | "filled" | "cancelled";

// Constants: UPPER_SNAKE_CASE
const MAX_RETRY_COUNT = 3;
const DEFAULT_PAGE_SIZE = 20;

// Utility functions: camelCase
function formatAddress(address: string): string { ... }
function parseTokenAmount(amount: bigint, decimals: number): string { ... }

// Event handlers: handleNoun or onNoun
function handleMintClick() { ... }
function onOrderSubmit(data: OrderFormData) { ... }
```

### Component Patterns

```tsx
// Props type defined above or inline for simple components
interface AssetCardProps {
  asset: PassportData;
  isLoading?: boolean;
  onSelect?: (tokenId: bigint) => void;
}

// Named export (not default export)
export function AssetCard({ asset, isLoading = false, onSelect }: AssetCardProps) {
  // 1. Hooks first
  const { address } = useAccount();
  const { data: ownerOf } = useOwnerOf(asset.tokenId);

    return <AssetCardSkeleton />;
  }

  // 4. Render
  return (
    <div className="rounded-lg border border-zinc-800 bg-zinc-900 p-4">
      {/* ... */}
    </div>
  );
}
```

### Hook Patterns

```typescript
// Contract read hook
export function usePassportData(tokenId: bigint) {
  return useReadContract({
    address: CONTRACTS.AssetPassport,
    abi: ASSET_PASSPORT_ABI,
    functionName: "getPassportData",
    args: [tokenId],
    query: {
      enabled: tokenId > 0n,  // Don't query for invalid IDs
    },
  });
}

// Contract write hook with full lifecycle
export function useMintPassport() {
  const { writeContract, data: hash, isPending, error } = useWriteContract();
  const {
    isLoading: isConfirming,
    isSuccess,
    data: receipt,
  } = useWaitForTransactionReceipt({ hash });

  const mint = useCallback(
    (to: `0x${string}`, name: string, metadataURI: string) => {
      writeContract({
        address: CONTRACTS.AssetPassport,
        abi: ASSET_PASSPORT_ABI,
        functionName: "mintPassport",
        args: [to, name, metadataURI],
      });
    },
    [writeContract]
  );

  return {
    mint,
    isPending,       // Waiting for user to sign
    isConfirming,    // Waiting for tx confirmation
    isSuccess,       // Tx confirmed
    error,           // Any error
    hash,            // Tx hash (available after signing)
    receipt,         // Full receipt (available after confirmation)
  };
}
```

### TypeScript Rules

- **Strict mode**: Always enable `strict: true` in tsconfig.json
- **No `any`**: Use `unknown` if type is truly unknown, then narrow with type guards
- **No `as` casts**: Use type predicates or discriminated unions instead
- **No non-null assertions**: Use optional chaining and nullish coalescing
- **Explicit return types**: Required for hooks and exported functions, optional for components

```json
// tsconfig.json
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "forceConsistentCasingInFileNames": true,
    "noUnusedLocals": true,
    "noUnusedParameters": true
  }
}
```

## Git Conventions

### Branch Naming

```
main                    # Production branch
develop                 # Development branch (optional)
feature/add-marketplace # New feature
fix/wallet-connection   # Bug fix
chore/update-deps       # Maintenance
refactor/hook-pattern   # Code restructuring
```

### Commit Messages

Format: `type(scope): description`

```
feat(contracts): add AssetManager with mint and transfer
feat(frontend): add marketplace page with order book
fix(hooks): handle undefined address in usePassportData
test(contracts): add edge case tests for order cancellation
chore(deps): update wagmi to v2.14
refactor(components): extract shared AddressDisplay component
docs: add deployment instructions to README
style: apply design system colors to sidebar
```

Types: `feat`, `fix`, `test`, `chore`, `refactor`, `docs`, `style`, `perf`, `ci`

Scope: `contracts`, `frontend`, `hooks`, `components`, `config`, `deploy`, or specific feature name

### What to Commit

- DO commit: source code, configs, tests, docs, deployment scripts, ABI files
- DO NOT commit: `.env.local`, `node_modules/`, `artifacts/`, `cache/`, `out/`, `.next/`, private keys

## Testing Expectations

### Contract Tests

- **Minimum**: Every public/external function has at least one happy-path test and one revert test
- **Target**: 90%+ line coverage on contracts
- **Required**: Test event emissions, state changes, access control
- Run with: `npx hardhat test`
- Coverage with: `npx hardhat coverage`

### Frontend Tests

- **Minimum**: Component render tests for all pages
- **Recommended**: Hook tests with mocked contract calls
- **Nice to have**: E2E tests with Playwright
- Run with: `npm run test` (vitest)

### Test File Location

- Contract tests: `test/ContractName.test.ts`
- Component tests: `src/components/**/__tests__/ComponentName.test.tsx`
- Hook tests: `src/hooks/__tests__/useHookName.test.ts`
- E2E tests: `test/e2e/*.spec.ts`

## Security Checklist

Before deployment, verify:

### Contracts
- [ ] All external/public functions have access control where needed
- [ ] ReentrancyGuard on functions with external calls
- [ ] Checks-Effects-Interactions pattern followed
- [ ] No floating pragma (locked to specific version)
- [ ] No `tx.origin` authentication
- [ ] All inputs validated (zero address, zero amount, bounds)
- [ ] Custom errors used instead of require strings
- [ ] Events emitted for all state changes
- [ ] No unused code or imports
- [ ] OpenZeppelin contracts used where possible (battle-tested)

### Frontend
- [ ] No private keys in client-side code
- [ ] No `any` types
- [ ] All async operations have error handling
- [ ] User input sanitized before contract calls
- [ ] Transaction amounts validated before submission
- [ ] Chain ID checked before transactions (prevent wrong network)
- [ ] Proper error messages for common failures (wrong chain, insufficient funds, rejected tx)

### Configuration
- [ ] `.env.local` is in `.gitignore`
- [ ] `.env.example` has all variables documented
- [ ] No secrets committed to git history
- [ ] RPC URLs point to correct network
- [ ] Contract addresses match deployed contracts

### Common Findings (Recurring Patterns)

- **Input validation gaps** — Missing zero-address checks, unbounded array inputs, missing amount bounds
- **Access control missing on admin functions** — State-changing functions without `onlyOwner` or role-based guards

- Apply Checks-Effects-Interactions on every function with external calls

## Regulatory & Jurisdiction Awareness

### UAE (Integra Home Base)

| Regulator | Scope |
|-----------|-------|
| DIFC / DFSA | Financial free zone, securities tokenization framework |
| VARA | Virtual assets regulation (Dubai) |
| DLD | Dubai Land Department — free API for real estate data |

### Compliance Requirements

- **ERC-7943 primitives are mandatory** for any contract handling tokens — `isTransferAllowed()`, `isUserAllowed()`, `forceTransfer()`, `setFrozen()`
- All RWA tokens must enforce compliance at the contract level, not just the frontend
- Jurisdiction checks must be on-chain or oracle-verified, not client-side only
